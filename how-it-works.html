The core idea

Dump a source into an inbox. A slash command reads it, writes a summary page, spawns or updates entity and concept pages, and stitches everything together with [[wikilinks]]. Over time you get a graph: people link to organizations, organizations link to sources, sources link back to the concepts they discuss.

Division of labor. The human picks sources and directs exploration. The LLM does the reading, writing, linking, and health-checking. No manual markdown edits needed.

Directory layout

CLAUDE.md       Schema and conventions (the LLM's rulebook)

Page types

source

Summary of an ingested document. Claims, quotes, metadata. Named src-topic-yyyy.md.

entity

A person, org, tool, or place. Named after itself, kebab-case.

concept

An idea, method, or framework. Named after itself, kebab-case.

analysis / comparison

Synthesis across multiple sources. Descriptive title.

Every page gets YAML frontmatter: title, type, created, updated. Source pages also track source_file, author, and date of the original.

Workflows

Ingest — turn a document into wiki pages

  1. Read the source in raw/inbox/ fully.
  2. Create a source summary page with claims and quotes.
  3. Update or create entity pages for the people and orgs mentioned.
  4. Update or create concept pages for the key ideas.
  5. Add [[wikilinks]] everywhere, in both directions.
  6. Update wiki/index.md and append to wiki/log.md.
  7. Move the file from raw/inbox/ to raw/.

A single source typically touches 5–15 pages.

Query — ask a question

  1. Read index.md to find relevant pages.
  2. Read those pages.
  3. Synthesize an answer with [[wikilinks]] citations.
  4. Offer to save substantial answers as a new analysis page via /save.

Fetch news — parallel agent teams

The /ufo-news command runs a six-phase protocol that parallelizes the expensive parts and serializes the dangerous parts:

  1. Fan out search agents — one per search track, collect candidate URLs.
  2. Dedupe against existing raw/ and index.md. Present to user.
  3. Fan out fetch agents — each writes its own file to raw/inbox/.
  4. Fan out analysis agents — read-only, return structured output.
  5. Serial merge — only the main agent writes to shared files.
  6. Report — surface cross-connections between new sources.
Why the split? Subagents have no shared memory and can race on shared files. Read and fetch in parallel, write serially.

Lint — health check

  • Find contradictions between pages.
  • Flag stale claims superseded by newer sources.
  • Identify orphan pages (no inbound links).
  • Spot concepts mentioned but lacking their own page.
  • Suggest missing cross-references and follow-up questions.

Conventions worth knowing

  • One entity per page. Split rather than merge.
  • Sources are immutable after ingest. New info means a new source page, not an edit.
  • Claims are attributed inline: According to [[src-paper-name]], ...
  • Contradictions are recorded explicitly, not silently resolved.
  • Bold key terms on first mention within a page.
  • Every page earns its links — both inbound and outbound. Dense graph is the goal.

The log format

## [YYYY-MM-DD] action | description

Details of what happened.

Actions are one of: ingest, query, lint, update, init. The log is append-only and parseable.
